# Coastal Wave Coupling Architecture

To add SnapWave + infragravity-wave physics for coastal Study Locations (Marshfield, San Francisco) without destabilising the working surge-only Marshfield batch, we build a parallel `build_sfincs_quadtree_snapwave.ipynb` notebook on a quadtree grid; compile SFINCS from the mt_Faber research branch and ship it as a side-by-side Apptainer SIF on the cluster; and force SnapWave boundaries from ERA5 ocean wave variables, with non-historical events drawing their wave forcing from the same historical analog used to supply their water-level forcing. Together these decisions create a separate `Wave-Coupled Truth Set` artefact path that future readers might otherwise expect to be merged with the regular-grid path — they are not, and this ADR records why.

## Considered options

**Notebook structure.** A — refactor the existing notebook to a quadtree-only path (rejected: blast-radius into the working surge-only Marshfield artefacts that the Event Catalog and GNS training depend on). B — keep the regular grid and hand-write SnapWave files outside hydromt-sfincs (rejected: violates the "official HydroMT syntax only" rule recorded in user memory; the installed hydromt-sfincs 2.0.0rc1 exposes SnapWave only through the quadtree code path). C — accepted: parallel notebook; the two artefact layouts diverge by design.

**Binary distribution.** Upstream PRs to hydromt-sfincs and wait for an official `deltares/sfincs` release that exposes IG waves and runup gauges (rejected: timeline unbounded). Modify the existing `deltares/sfincs-cpu:sfincs-v2.3.2` image (rejected: that image lacks the SnapWave_IG branch source). Accepted: build mt_Faber from source on DSAI and package side-by-side as a new Apptainer SIF; select per-job via `SFINCS_IMAGE`.

**Wave forcing for non-historical events.** (i) Wave-free siblings — run stochastic / MSL-shift events without SnapWave (rejected by domain expert: biases peak water-level magnitudes across the catalog by mixing wave-and-no-wave physics). (ii) Conditional sampling from ERA5 climatology (rejected: requires a defensible conditional sampler we do not have). (iii) Empirical Hm0(surge) regression (rejected: loses temporal shape of the spectrum). (iv) JPM-OS-style co-simulation from a parametric wind field (rejected: Direct AORC SST rainfall members do not carry wind fields). Accepted: **joint analog resampling** — the same historical analog that supplies water-level forcing to a non-historical event also supplies the ERA5 wave forcing for that event. This preserves observed wave-surge correlation, requires no new modelling, and is upgradeable to JPM-OS later if synthetic wind fields become available. Aligns with the principle in Resio & Westerink (2008) and the JPM-OS literature of driving surge and waves from the same atmospheric driver.

## Consequences

- `build_sfincs_quadtree_snapwave.ipynb` (new) and `build_sfincs_generic.ipynb` (existing) write to **different artefact layouts**. A `Wave-Coupled Truth Set` carries a quadtree mesh, SnapWave boundary points, and runup-gauge outputs; a regular-grid `Hydrodynamic Truth Set` does not. Downstream consumers (Subgrid Flood Layer extractors, Evaluation Layer builders, GNS preprocessors) must branch on the `Coastal Wave Coupling` flag declared in each `locations/<study_location>/config.yaml`.
- The cluster carries two SFINCS images going forward: `sfincs-cpu_latest.sif` (v2.3.2, surge-only) and `sfincs-snapwave_latest.sif` (mt_Faber, wave-coupled). Slurm scripts select per-job by env var.
- Runup gauges and the `rugfile` config key are not yet wrapped by hydromt-sfincs 2.0.0rc1. They are written by thin helpers in `src/sfincs_runs/snapwave_setup.py` using the **official SFINCS file/config syntax** (documented in the mt_Faber SFINCS source), with the config key injected into `sfincs.inp` after `sf.write()`. This is the one deviation from "use hydromt-sfincs schema only"; it is bounded to a single helper and is upstreamable as a hydromt-sfincs PR later. The current mt_Faber source checks `obsfile` before opening `rugfile` in `read_rug_file`, so the helper also creates a tiny observation file when no real observation file is configured.
- The Event Catalog gains an implicit constraint: every non-historical event must carry a historical-analog reference that resolves to both a water-level segment and an ERA5 wave segment from the same storm-window. Catalog builders for Stochastic Storm Transposition Events and MSL-Shift Scenarios must record this analog explicitly.
