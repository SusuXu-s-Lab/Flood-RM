# Flood-RM Codebase Map

Generated by a read-only audit pass. Detailed evidence lives in
`docs/audit/file_inventory.json`.

## Assumptions

- The source of truth for user-facing workflow order is each `locations/<name>/pipeline.py`
  plus the notebooks under `locations/<name>/`.
- Generated datasets under `locations/*/data/` are not source code and were not inventoried
  except where notebooks refer to their paths.
- "Oversized" means a Python file at or above roughly 800 lines, not a defect by itself.
- Category labels are planning labels, not edit permission.

## Repository Shape

- `README.md` describes two main products: a synthetic Grid Dataset for dynamic
  microgrid studies, and stochastic flood-driver catalogs sampled into SFINCS,
  SnapWave, or Wflow-SFINCS scenarios.
- `pyproject.toml` packages `design_events`, `sfincs_runs`, `wflow_runs`, `power`,
  and `fiat_runs`, with `study_location.py` force-included as a top-level module.
- `scripts/` contains workflow tools: notebook pipeline execution, SMART-DS download,
  structure acquisition, and PowerModelsONM/DynaGrid bundle export.
- `locations/` contains four Study Locations or partial Study Locations: `marshfield`,
  `austin`, `greensboro`, and `sfo`.
- `src/` contains 131 Python files across about 63,278 lines, including `scripts/`.
- `locations/` contains 43 notebooks outside generated data folders.

## Classification Summary

- `CORE_SCIENCE`: 74 files.
- `NOTEBOOK_API`: 46 files.
- `PLUMBING_IO`: 30 files.
- `VALIDATION_QA`: 8 files.
- `PLOTTING_REPORTING`: 5 files.

Overlap is expected. For example, `src/sfincs_runs/build_base/region_notebook.py` is
both `CORE_SCIENCE` and `NOTEBOOK_API`, while also containing replaceable plotting and
path plumbing.

## High-Level Architecture Friction

1. Notebook-facing modules are often shallow: their interface exposes many small
   functions and callers must understand artifact layout, config defaults, plotting,
   validation, and model semantics together.
2. Several modules mix science and plumbing. Examples include `region_notebook.py`,
   `collect_sources/workflow.py`, `wflow_runs/build_plan.py`, and
   `power/resilience/switches.py`.
3. Plotting modules are very large and carry repeated Matplotlib setup logic. This is
   likely replaceable by thinner plotting helpers plus stable data-preparation functions.
4. Path/config/manifest handling repeats across `study_location.py`, `scripts/`,
   `collect_sources`, `sfincs_runs`, `wflow_runs`, `power.exports`, and `fiat_runs`.
5. Existing tests are broad enough to support incremental cleanup, but refactors should
   first add characterization tests around notebook-facing imports and pure science
   functions.

## Shared Paths And Study Location Helpers

The first shared-infrastructure refactor kept `src/study_location.py` as the public
notebook-facing facade while moving helper implementations into smaller modules. The
follow-up path cleanup replaced `_study_location_paths.py` with `src/paths.py`, a shared
path helper used by `study_location.py`, `design_events.runtime`, `sfincs_runs.config`, and
power runtime/export modules. The shared path helper is deliberately small and
predictable: explicit relative paths resolve under the repository root instead of
probing the current working directory first.

- `src/paths.py`: minimal repository/location path resolution and GeoJSON IO.
- `src/aoi.py`: short AOI builder and study-area bounding-box helper.
- `src/study_location.py`: public facade plus the remaining Study Location identity
  resolver and explicit location-YAML merge. The old starter-template/listing helpers,
  hidden methodology defaults, and recipe-inheritance helper module were deleted because
  existing YAML files and direct runtime resolution made them unnecessary.

`study_location.py` no longer applies hidden methodology defaults. It merges only
explicit location YAML and included detail files; source/model defaults should live in
the domain modules that own them or in the location YAML when they are location choices.
The split is not a mandate to keep private helper modules forever; the next integration
pass should reduce repeated path/runtime conventions across domains where behavior
matches.

## Power Artifact Helper Reduction

The power-domain artifact cleanup deleted `src/power/artifacts.py` and the temporary
private `_artifact_*` split modules. No notebook-facing imports depended on
`power.artifacts`; remaining source callers now use `src/paths.py`, `study_location.py`,
or module-local parsing/ID helpers where the artifact contract is actually owned.

No SSAP, topology, SMART-DS-compatible schema, ONM export, or Block Invariant Contract
behavior changed. Generic parquet writing, optional SHA wrappers, validation-error
appenders, and one-off Stable Grid ID functions were removed from the public artifact
surface. The SMART-DS-compatible export now keeps schema-aware PyArrow writes, file
hashing, manifest hashing, and asset/control-unit ID construction local to
`src/power/exports/smart_ds_grid.py`.

The Marshfield grid notebooks no longer depend on `src/power/notebook.py`; that shallow
facade was deleted. The notebooks now import `sfincs_runs.config.load_runtime` and
`sfincs_runs.config.build_grid_paths` directly, then create the same Grid Dataset output
directories in their setup cell.

## SFINCS Plumbing Reduction

The SFINCS cleanup deleted shallow helper modules after migrating their callers:

- `src/sfincs_runs/notebook.py`: notebooks now import `sfincs_runs.config.load_sfincs_runtime`.
- `src/sfincs_runs/scenarios/io.py`: generic JSON writing was inlined; SFINCS input parsing lives in `sfincs_runs.config.parse_sfincs_inp`; event-only `sfincs.inp` and `sfincs.bzs` writers are private to `scenarios.event_forcing`.
- `src/sfincs_runs/build_base/region_setup.py`: `RegionSetup` and `build_region_setup` now live in `build_base.static_intake`, where the static-intake paths are consumed.
- `src/sfincs_runs/single_use_case/`: the example-run plan is private to `scenarios.event_forcing.build_event`, its only source caller.

Do not recreate generic SFINCS IO, notebook, or single-use facades unless a notebook-facing contract needs a documented compatibility layer.
